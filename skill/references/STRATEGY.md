
# Стратегия исследования в Mattermost

Mattermost — разговоры, не документация. 60%+ решений в комментариях, не в корневых постах.

## Принципы

### 0. Максимально короткая строка для поиска

Хорошо работают запросы из 1-2 слов, можно попробовать просклонять русские слова для лучшего поиска.

### 1. Углубляйся в треды

Нашёл релевантный пост → ВСЕГДА скачай тред целиком (`get_post_thread.py`).
Нашел комментарий через поиск — запроси весь тред чтобы понять контекст. Если в треде есть релевантные ссылки на другие треды — возьми их в исследования.
Если тред нерелевантен и не содержит релевантных запросу комментариев - не переходи по ссылкам.

Признаки важного треда: `reply_count > 5`, участники из разных команд, слова "решили"/"итого"/"зафиксировали".
Реакции `:done3:`, `:plus1:` часто маркируют принятые решения — обращай на них внимание.

### 2. Следуй за ссылками — особенно в корневом посте

**Если ссылка находится в корневом посте треда или напрямую касается обсуждаемой темы — переходи по ней ОБЯЗАТЕЛЬНО, не спрашивая.** 
Корневой пост задаёт контекст всего обсуждения, и ссылки в нём — первоисточник, без которого нельзя полноценно ответить на вопросы в треде.

Встретил в релевантном сообщении ссылку:
- `https://mm.../pl/{post_id}` → извлеки post_id, запроси тред
- Confluence/Jira URL → **если в корневом посте или по теме — сразу запроси через соответствующий скилл/API/MCP**; если косвенная — зафиксируй как источник
- Git → зафиксируй как артефакт, если нужно воспользуйся соответствующим скиллом/CLI/MCP чтобы понять есть ли там релевантное содержимое

### 3. Анализируй прикреплённые файлы

Если есть файлы в релевантном посте или сообщении — скачай их чтобы посмотреть

Есть `file_ids` или `files` → сначала `get_file_info.py`:
- `.puml`, `.archimate`, `.drawio`, `.pdf`, `.docx` → **скачивай** — схемы и спеки
- `.png`, `.jpg` → скачивай и получай расшифровку
- `.zip`, `.jar`, `.log` → обычно не нужны

### 4. Ищи по сообщениям людей, расширяй поиск через них

Если знаешь что у конкретного человека есть компетенция - ищи по сообщениям людей (никнейму, фамилия, идентификатор, email).
Часто в его сообщениях в других тредах есть ответы.
Если знаешь что нужную/полезную статью/заметку/комментарий написал конкретный человек - поищи по нему.
Если нашел релевантное сообщение — углубляйся в тред.
Нашёл ключевого участника → `from:username` → его посты → новые каналы, люди, темы.
Не углубляйся больше 2 уровней (человек → его посты → новые люди — стоп).

### 5. Permalink'и к каждому факту

Каждый вывод из переписки подкрепляй permalink'ом вида `https://{MATTERMOST_URL}/{team_name}/pl/{post_id}`.

- `{team_name}` — это поле `name` команды (slug), а НЕ channel name и НЕ team_id. Получить: `python get_my_teams.py` → брать `name` (например, `platforma-global`).
- Для текущего инстанса: `https://mm.podeli.ru/platforma-global/pl/{post_id}`.
- Формат `/channels/pl/{post_id}` и `/channel/pl/{post_id}` — **не работают**, вернут 404.

⚠️ **Permalink строится ТОЛЬКО по `post_id` (поле `id` поста).** Частая ошибка — подставить вместо него:
- `channel_id` (особенно если дамп сохранён в файл `{channel_id}.json` — имя файла это НЕ id поста);
- `root_id` в сокращённом виде или первые символы id;
- id пользователя / файла / реакции.

Все эти варианты вернут 404 молча. Если у треда нет реплаев — permalink всё равно указывает на `id` корневого поста (а не на канал). Перед публикацией ссылки убедись, что id взят из поля `id` конкретного поста в ответе API, длина 26 символов, и это не `channel_id` треда, который ты только что читал.

### 6. Если вопрос требует общих технических знаний используй Perplexity MCP

Вопросы архитектурного характера, технического, не относящиеся к самописным сервисам или конфигурации сервисов внутри компании - 
ищи в интернете (например perplexity mcp)

## Шаблоны по типу задачи

### Профиль человека

Чтобы определить компетенцию человека:

1. Каналы где активен → область ответственности
2. Раскрыть ключевые треды

### Если требуется знания по сервису / компоненту / интеграции

Если сервис компании или его конкретная конфигурация внутри компании:

1. Искать название сервиса (рус/англ)
3. Раскрыть все треды
4. Определить ключевых людей → поиск `from:` по ним
5. Подкрепить данными  (RFC, спеки, схемы) в том числе из других источников (confluence / git / archimate) при помощи соответствующих скиллов/API/MCP
6. Если необходимо скачивай репозитории сервисов, их деплойменты helm

Если общий вопрос (general knowledge):

1. используй доступные инструменты поиска

## Если какая-то информация добывалась во множестве запросов и это можно было сократить используя стратегию — оповести пользователя.
